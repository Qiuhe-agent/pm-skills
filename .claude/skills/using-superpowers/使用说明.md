# Using Superpowers 技能使用说明

## 技能简介

`using-superpowers` 是一个核心技能，它的主要作用是帮助你更有效地发现和使用其他技能。这个技能会强制你在执行任何任务之前先检查是否有相关的技能可用。

## 核心原则

### 1. 技能使用强制规则

> **如果认为有 1% 的可能性某个技能适用，就必须使用该技能！**

这不是可选的，这是必须遵守的规则。

### 2. 指令优先级

1. **用户的明确指令**（CLAUDE.md、GEMINI.md、AGENTS.md、直接请求）- 最高优先级
2. **Superpowers 技能** - 覆盖默认系统行为
3. **默认系统提示** - 最低优先级

示例：如果 CLAUDE.md 说"不要使用 TDD"，而某个技能说"始终使用 TDD"，请遵循用户的指令。用户拥有控制权。

## 工作流程

```
用户消息收到
        ↓
   可能有技能适用？
   /              \
是（即使 1%）    绝对不适用
  ↓                    ↓
调用 Skill 工具      回复（包括澄清）
  ↓
宣布："使用 [skill] 来 [purpose]"
  ↓
有检查清单吗？
   /      \
  是       否
  ↓        ↓
为每个项目创建 TodoWrite 任务
  ↓
严格遵循技能
```

## 如何访问技能

### 在 Claude Code 中

使用 `Skill` 工具。当你调用一个技能时，它的内容会被加载并呈现给你 —— 直接遵循它即可。永远不要使用 Read 工具读取技能文件。

### 在 Gemini CLI 中

技能通过 `activate_skill` 工具激活。Gemini 在会话开始时加载技能元数据，并按需激活完整内容。

### 在其他环境中

请查看你平台的文档了解技能是如何加载的。

## 平台适配

技能使用 Claude Code 工具名称。非 CC 平台：请参阅 `references/codex-tools.md`（Codex）了解工具等效项。Gemini CLI 用户通过 GEMINI.md 自动加载工具映射。

## 危险信号（Red Flags）

以下想法意味着你需要停下来 —— 你在合理化：

| 想法 | 现实 |
|------|------|
| "这只是一个简单的问题" | 问题就是任务。检查技能。 |
| "我先需要更多上下文" | 技能检查在澄清问题之前。 |
| "让我先探索一下代码库" | 技能告诉你如何探索。先检查。 |
| "我可以快速检查 git/文件" | 文件缺乏对话上下文。检查技能。 |
| "让我先收集信息" | 技能告诉你如何收集信息。 |
| "这不需要正式的技能" | 如果技能存在，就使用它。 |
| "我记得这个技能" | 技能会演变。阅读当前版本。 |
| "这不算任务" | 行动 = 任务。检查技能。 |
| "这个技能太小题大做了" | 简单的事情会变复杂。使用它。 |
| "我先做这一件事" | 在做任何事之前先检查。 |
| "这感觉很有成效" | 无纪律的行动浪费时间。技能防止这一点。 |
| "我知道那是什么意思" | 知道概念 ≠ 使用技能。调用它。 |

## 技能优先级

当多个技能可能适用时，按以下顺序使用：

1. **流程技能优先**（头脑风暴、调试）- 这些决定如何处理任务
2. **实现技能第二**（前端设计、mcp-builder）- 这些指导执行

示例：
- "让我们构建 X" → 先头脑风暴，然后实现技能
- "修复这个 bug" → 先调试，然后领域特定技能

## 技能类型

- **严格型**（TDD、调试）：严格遵循。不要为了适应而放弃纪律。
- **灵活型**（模式）：根据上下文调整原则。

技能本身会告诉你是哪种类型。

## 用户指令

指令说"做什么"，而不是"怎么做"。"添加 X"或"修复 Y"并不意味着跳过工作流。

---

## 快速开始

1. 收到用户消息后，先暂停
2. 问自己："有没有可能适用的技能？"
3. 如果答案是"是"（即使只有 1% 的可能性），使用 `Skill` 工具调用该技能
4. 宣布你正在使用该技能
5. 严格遵循技能的指示
6. 然后再回复用户

记住：**技能检查在任何回复或行动之前！**
