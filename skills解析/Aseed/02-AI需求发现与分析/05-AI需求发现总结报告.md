**版本：** v1.0

**状态：** 待验证

---

## 执行摘要

本报告总结了Aseed AI运维产品的需求发现阶段成果。通过用户画像、问题陈述、JTBD分析和问题框架画布，我们系统地理解了目标用户的痛点、需求和期望。

### 核心发现
1. **告警风暴是真实存在的P0问题**——中大型企业日均告警5,000+条，90%是噪音
2. **根因定位是最大痛点**——平均1-3小时才能找到根因，像"大海捞针"
3. **重复故障反复发生**——80%的故障是重复的，但缺乏自动化手段
4. **团队压力巨大**——On-call频繁，离职率高，生活质量受影响
5. **AI接受度高但有顾虑**——愿意尝试AI，但需要可解释性和人工兜底

- 完成5家客户POC验证（Q2 2026）

## 目标用户总结

### 四个核心用户角色

| 角色 | 痛点 | 核心需求 | 决策影响力 |

### 用户共识
- **都认同：** 告警太多、根因定位太难、重复故障太多

- **都担心：** AI不可靠、数据不安全、工具太复杂

## 核心痛点总结

### Pain Point 1：告警风暴 🔥🔥🔥

**现状：**
- 日均告警量：5,000-20,000条
- 有效告警占比：<10%
- 结果：真正严重的问题被淹没，发现延迟

**用户原话：**
> "每天早上第一件事就是删告警，删到手指酸。"
>
> "告警太多了，干脆不看了，结果有一次真的错过了重要告警。"

**为什么这是P0问题？**
- 直接影响故障发现时间
- 造成团队疲劳和麻木
- 是其他问题的根源

---

### Pain Point 2：根因定位困难 🔥🔥🔥

**现状：**
- 平均根因定位时间：1-3小时
- 需要跨多个工具：监控、日志、调用链、CMDB、ITSM
- 依赖专家经验：资深工程师才能快速定位

**用户原话：**
> "排查问题像大海捞针，这个看看，那个翻翻，时间就过去了。"
>
> "要是有工具能直接告诉我'因为X改了，导致Y发生了'就好了。"

**为什么这是P0问题？**
- 直接影响MTTR（故障恢复时间）
- 知识传承困难
- 专家成为瓶颈

---

### Pain Point 3：重复故障频发 🔥🔥🔥

**现状：**
- 80%的故障是重复发生的
- 自动化覆盖率：<20%
- 同样的坑反复踩

**用户原话：**
> "同样的OOM故障，这个月已经发生5次了。"
>
> "每次都是手动重启，根本问题还是没解决。"

**为什么这是P0问题？**
- 团队陷入"救火"循环
- 没时间做长期优化
- 挫败感强

---

### Pain Point 4：知识传承困难 🔥🔥

**现状：**
- 新人上手时间：3个月
- 知识都在人脑子里：资深工程师离职，经验就带走
- 没有系统化沉淀

**用户原话：**
> "带一个新人要3个月才能独立干活，太累了。"
>
> "上次老张离职，很多问题就没人知道怎么处理了。"

---

### Pain Point 5：On-call压力大 🔥🔥

**现状：**
- 每周夜间被叫：2-3次
- 睡眠质量差：On-call那天根本睡不好
- 团队离职率：30%+

**用户原话：**
> "On-call那天我手机都握在手里，一响心就跳。"
>
> "我老婆说我最近脾气特别差，都是On-call闹的。"

## 核心需求总结

### Must-Have（必须有）

| 需求 | 说明 | 为什么必须有？ |
|------|------|---------------|
| **告警降噪** | 从5,000条降到500条，只看重要的 | 不解决这个，其他都免谈 |
| **根因分析** | 直接告诉用户"为什么" | 这是最大的痛点 |
| **自动修复** | 80%的常见问题不用人管 | 从"救火"转向"防火" |
| **集成现有工具** | Prometheus、ELK、Jaeger等 | 客户不会替换现有工具 |
| **数据安全** | 支持私有化部署 | 硬性要求，没有就不用谈 |

### Should-Have（应该有）

| 需求 | 说明 |
|------|------|
| **预测性告警** | 在故障发生前发现风险 |
| **知识沉淀** | 经验不随人走 |
| **量化成果** | 有数据向领导汇报 |
| **运维Copilot** | 指导新人排查问题 |

### Could-Have（可以有）

| 需求 | 说明 |
|------|------|
| **自动生成复盘报告** |
| **容量规划建议** |
| **变更风险评估** |

## AI特有考虑

### 用户对AI的态度

| 态度 | 比例 | 说明 |
|------|------|------|
| **积极拥抱** | 30% | 愿意尝试新技术，相信AI能帮助 |
| **谨慎乐观** | 50% | 愿意尝试，但要看实际效果 |
| **怀疑观望** | 20% | 担心AI不可靠，再看看 |

### AI需求

| 需求 | 说明 | 刚性程度 |
|------|------|---------|
| **AI决策必须可解释** | 不能是"黑盒"，要知道为什么 | 🔴 硬性 |
| **需要人工兜底机制** | AI建议，人工确认，不能全自动 | 🔴 硬性 |
| **AI"幻觉"需要避免** | 不能误导用户 | 🟠 中等 |
| **模型需要持续学习** | 不是"一次训练，永久使用" | 🟠 中等 |

### 信任建立路径

1. **阶段1：AI辅助**——AI给出建议，人工决策
2. **阶段2：AI推荐**——AI推荐方案，人工确认
3. **阶段3：半自动**——常见问题AI自动处理，异常情况人工介入
4. **阶段4：全自动**——完全信任AI，但仍有紧急停止按钮

## 约束与边界

### 技术约束

| 约束 | 说明 |
|------|------|
| 必须和现有工具集成 | Prometheus、ELK、Jaeger、Kubernetes等 |
| 不能替换现有工具 | 只能增强，不能替换 |
| 数据不能出客户网络 | 支持私有化部署 |
| 部署时间≤1周 | 不能需要数月部署 |
| 培训时间≤1天 | 不能需要大量培训 |

### 边界：什么不做？

- ❌ 不做完整的监控系统（只做增强）
- ❌ 不做ITIL/ITSM系统（只做集成）
- ❌ 不做CMDB系统（只做集成）
- ❌ 不做底层基础设施（只做智能层）
