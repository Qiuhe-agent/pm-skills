## Persona 1：运维张伟 - 运维经理

### 基本信息
- **姓名：** 运维张伟（Ops Zhang Wei）
- **年龄：** 38岁
- **职位：** 某股份制银行数据中心运维经理
- **团队规模：** 25人
- **从业年限：** 12年
- **地理位置：** 上海
- **家庭状况：** 已婚，有一个8岁的孩子
- **在线活跃度：** LinkedIn活跃，经常参加运维技术大会

### 个人画像
> "我每天早上醒来第一件事就是看手机，有没有夜间的告警。这个行业干了12年，感觉自己越来越像个'救火队员'，而不是一个'管理者'。"

### 典型的一天
- 8:00 - 到公司，先看夜间告警汇总
- 9:00 - 团队站会，分配今天的任务
- 10:00 - 处理昨天遗留的故障复盘
- 11:00 - 和业务部门开会，讨论下个季度的稳定性保障
- 14:00 - 评审新系统上线方案
- 16:00 - 面试（团队又有人离职了）
- 18:00 - 准备下班，心里有点不踏实
- 21:00 - 手机告警响了，远程处理一个小问题
- 23:00 - 终于可以睡觉了

### 他的痛点
1. **团队压力大，离职率高** - 去年团队走了5个人，On-call太累了
2. **领导要求"零故障"，但实际很难做到** - 业务部门只要结果，不管过程
3. **预算有限，难以扩充团队** - 申请了半年的headcount还没批下来
4. **被告警淹没，分不清轻重缓急** - 每天5,000+条告警，90%都是噪音
5. **知识传承困难** - 资深工程师离职后，很多经验就带走了

### 他想要实现什么
1. **降低团队工作强度** - 让大家能正常下班，周末能陪陪家人
2. **提升故障处理效率** - 故障发生时能快速定位和解决
3. **有数据向领导汇报工作成果** - 不能只说"我们很努力"，要说"我们把MTTR降低了X%"
4. **建立知识库，沉淀运维经验** - 就算有人离职，经验还在
5. **从"救火"转向"防火"** - 能在故障发生前就发现问题

### 他的决策影响力
- **采购决策权：** 有预算权限到¥100万，超过需要总监审批
- **决策影响者：** 他的团队成员、安全合规部门、IT总监
- **信任的信息源：** Gartner报告、同行推荐、技术大会分享
- **对AI的态度：** 谨慎乐观，希望看到实际效果，不喜欢概念炒作

### 他的典型语录
> "我要的不是更多告警，而是更准的告警。"
>
> "工具再好，也要人能用。不要让我花3个月来培训团队。"
>
> "给我看数据，不是PPT。MTTR降了多少？告警少了多少？"
>
> "On-call那天我根本睡不好，手机一响心就跳。"

### 验证点
- [ ] 团队离职率是否真的这么高？
- [ ] 日均告警量是否真的有5,000+条？
- [ ] 预算权限是否准确？
- [ ] 对AI的态度是否如此？

## Persona 2：SRE李娜 - SRE团队负责人

### 基本信息
- **姓名：** SRE李娜（SRE Li Na）
- **年龄：** 32岁
- **职位：** 某电商平台SRE团队负责人
- **团队规模：** 12人
- **从业年限：** 8年
- **地理位置：** 杭州
- **家庭状况：** 已婚，无小孩
- **技术栈：** Kubernetes、Prometheus、Grafana、Terraform
- **在线活跃度：** GitHub活跃，Twitter/Tech Twitter重度用户

### 个人画像
> "我每天的时间都被告警占满了，根本没时间做那些真正能提升稳定性的长期优化工作。感觉我们就像在原地踏步。"

### 典型的一天
- 9:00 - 到公司，先看Grafana大盘
- 9:30 - 处理凌晨的告警（又是几个误报）
- 10:00 - 团队站会，讨论今天的告警
- 11:00 - 排查一个慢查询问题（花了2小时才找到原因）
- 13:00 - 午饭，和团队聊聊最近的技术趋势
- 14:00 - 审核新同学写的Playbook
- 15:00 - 本来想做容量规划，但又被告警打断了
- 17:00 - 准备大促保障方案（下个月就是618了）
- 19:00 - 下班，今天又没做多少"正事"

### 她的痛点
1. **每天被告警淹没，没时间做长期优化** - 70%的时间在处理告警
2. **新人上手慢，知识传承困难** - 带一个新人要3个月才能独立干活
3. **业务团队对稳定性要求越来越高** - "双11零故障"是硬性指标
4. **同样的故障反复发生** - 这个月OOM已经发生5次了
5. **排查问题像大海捞针** - 要看监控、翻日志、查变更，太耗时

### 她想要实现什么
1. **从"救火"转向"防火"** - 能在故障发生前就发现风险
2. **沉淀运维知识，减少对专家依赖** - 就算我离职，团队也能正常运转
3. **用数据驱动稳定性提升** - 知道哪里有问题，优先优化哪里
4. **同样的故障不要发生第二次** - 自动化修复常见问题
5. **有时间做真正有价值的工作** - 容量规划、架构优化、混沌工程

### 她的决策影响力
- **采购决策权：** 有推荐权，最终决策需要CTO审批
- **决策影响者：** 她的团队、运维经理、CTO
- **信任的信息源：** GitHub trending、Tech Twitter、行业技术博客
- **对AI的态度：** 积极拥抱，愿意尝试新技术，但要看实际效果

### 她的典型语录
> "我要的不是更多数据，而是更好的洞察。告诉我'因为X改了，导致Y发生了'。"
>
> "同样的故障发生第三次，我就觉得我们的工作没做好。"
>
> "我希望有一天，On-call只是个备份，而不是常态。"
>
> "给我API，让我能自己写脚本集成。不要做封闭系统。"

### 验证点
- [ ] 处理告警是否真的占70%时间？
- [ ] 对API和开放性的需求是否如此强烈？
- [ ] 大促压力是否真的这么大？

- **公司规模：** 800人
- **从业年限：** 18年

### 他的关注点
1. **系统稳定性** - 不能接受影响业务的故障
2. **运维成本控制** - 人力成本占IT支出比例越来越高
3. **团队建设** - 留住人才，提升团队能力
4. **业务支撑** - IT要能支撑业务快速发展
5. **风险管控** - 要有预案，不能"裸奔"

1. **稳定性事故影响业务和声誉** - 去年一次故障，公司股价跌了5%
2. **运维成本持续上升** - 运维团队从10人扩到30人，成本涨了3倍
3. **人才难招也难留** - 市场上资深SRE年薪要60万+，还不一定留得住
4. **业务发展太快，基础设施跟不上** - 业务要上线新功能，但运维保障能力不足

### 他想要实现什么
1. **提升系统稳定性** - 减少故障次数和故障时长
2. **控制运维成本** - 不能无限扩人，要靠工具提升人效
3. **支撑业务快速发展** - 业务能快速迭代，基础设施不能成为瓶颈
4. **建立风险管控体系** - 有预测、有预案、有复盘

### 他的决策影响力
- **采购决策权：** 最终决策者，预算¥500万以内自己拍板
- **决策影响者：** 运维负责人、财务负责人、CEO
- **信任的信息源：** Gartner、Forrester、同行CTO推荐
- **对AI的态度：** 务实，看重ROI，不介意多花钱但要有实际效果

### 他的典型语录
> "我不要听'AI'、'智能'这些概念，告诉我能把MTTR降多少，能省多少人力成本。"
>
> "每次故障，我都要向CEO解释。我希望这种解释越少越好。"

## 总结与验证计划

### 主要假设
1. **告警风暴是普遍问题** - 中大型企业日均告警量5,000+条
2. **On-call压力大** - 运维团队离职率高、睡眠质量差
3. **根因定位困难** - 平均1-3小时才能找到根因
4. **AI接受度高** - 运维团队愿意尝试AI辅助工具
5. **重复故障多** - 80%的故障是重复发生的

### 验证计划
- **Q1 2026：** 完成20家客户深度访谈
- **Q2 2026：** 完成5家客户POC，验证假设
- **Q3 2026：** 根据反馈更新用户画像
