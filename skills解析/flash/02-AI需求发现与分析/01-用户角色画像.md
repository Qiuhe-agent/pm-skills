## 画像一：运维经理（决策者）

### 基本信息

| 属性 | 描述 |
|------|------|
| **姓名** | 李明（化名） |
| **职位** | IT运维经理 / 运维总监 |
| **年龄** | 35-45岁 |
| **工作年限** | 10-15年 |
| **团队规模** | 管理10-50人运维团队 |
| **企业规模** | 500-5000人企业 |
| **行业** | 金融、互联网、制造业 |

### 个人背景

李明是一位经验丰富的运维管理者，从一线运维工程师成长起来，深知运维工作的痛点。他负责整个运维团队的日常管理和战略规划，需要向CTO/CIO汇报运维效率和系统稳定性指标。

### 典型工作场景

- **每日晨会**：审查前一天的告警处理情况、故障复盘
- **团队管理**：人员招聘、培训、绩效评估
- **预算规划**：运维工具采购、云资源成本控制
- **跨部门协作**：与开发团队协调发布计划，与业务部门沟通SLA
- **应急响应**：重大故障时协调资源、向上汇报

### 核心痛点

| 痛点 | 具体描述 | 痛苦程度 |
|------|---------|---------|
| 告警风暴 | 每天收到数千条告警，无法快速识别关键问题 | ⭐⭐⭐⭐⭐ |
| 人才流失 | 资深运维工程师难招难留，知识断层严重 | ⭐⭐⭐⭐⭐ |
| 效率低下 | 团队大量时间花在重复性工作上，效率难以提升 | ⭐⭐⭐⭐ |
| 汇报压力 | 需要向管理层证明运维团队的价值 | ⭐⭐⭐⭐ |
| 成本控制 | 运维预算有限，需要在效率和成本之间平衡 | ⭐⭐⭐⭐ |

### 目标与动机

**主要目标**：
- 提升运维团队整体效率，减少重复性工作
- 降低MTTR（平均恢复时间），提升系统稳定性
- 降低对资深专家的依赖，让新人快速上手
- 用数据证明运维团队的价值

**购买动机**：
- 解决告警噪音问题，提升团队响应效率
- 降低人力成本，减少对专家的依赖
- 提升团队技术能力，保持竞争力

### 决策因素

| 因素 | 权重 | 关注点 |
|------|------|--------|
| 效果验证 | 30% | 能否真正解决问题，有无可参考案例 |
| 成本效益 | 25% | ROI是否合理，投入产出比 |
| 实施难度 | 20% | 部署周期、学习成本、对现有系统的影响 |
| 安全合规 | 15% | 数据安全、合规认证 |
| 服务支持 | 10% | 售后服务、培训支持 |

### 典型语录

> "每天早上打开手机就是几百条告警，根本分不清哪些是关键问题。"
> 
> "招一个资深运维工程师太难了，好不容易培养出来又跳槽了。"
> 
> "我需要工具帮我证明运维团队的价值，而不是让我天天解释为什么又出故障了。"

## 画像二：SRE工程师（核心用户）

### 基本信息

| 属性 | 描述 |
|------|------|
| **姓名** | 张伟（化名） |
| **职位** | SRE工程师 / 高级运维工程师 |
| **年龄** | 28-35岁 |
| **工作年限** | 5-10年 |
| **技术栈** | Kubernetes、Prometheus、Grafana、Python |
| **企业规模** | 500-5000人企业 |
| **行业** | 互联网、金融科技 |

### 个人背景

张伟是一位技术能力强的SRE工程师，负责核心系统的稳定性保障。他熟悉云原生技术栈，对自动化运维有较高追求。他是团队中的技术骨干，经常参与故障排查和系统优化。

### 典型工作场景

- **值班响应**：7x24小时轮值，处理告警和故障
- **故障排查**：分析日志、指标、链路，定位根因
- **系统优化**：容量规划、性能调优、架构改进
- **工具开发**：开发自动化脚本、监控大盘
- **知识分享**：编写文档、团队培训

### 核心痛点

| 痛点 | 具体描述 | 痛苦程度 |
|------|---------|---------|
| 故障定位慢 | 多系统关联复杂，根因定位耗时长 | ⭐⭐⭐⭐⭐ |
| 重复性工作 | 大量时间处理相似问题，无法专注于优化 | ⭐⭐⭐⭐⭐ |
| 知识分散 | 故障处理经验没有沉淀，每次都要重新排查 | ⭐⭐⭐⭐ |
| 工具割裂 | 多个监控工具，数据分散，难以关联分析 | ⭐⭐⭐⭐ |
| 值班压力 | 夜间告警频繁，影响生活质量 | ⭐⭐⭐⭐ |

### 目标与动机

**主要目标**：
- 快速定位故障根因，减少故障恢复时间
- 自动化处理常见问题，减少重复性工作
- 建立运维知识库，提升团队整体能力
- 改善值班体验，减少无效告警打扰

**使用动机**：
- 提升故障排查效率，减少加班
- 学习AI技术，保持技术竞争力
- 减少重复性工作，专注于有价值的优化

### 使用场景

| 场景 | 具体需求 |
|------|---------|
| 告警处理 | 快速识别关键告警，获取根因分析 |
| 故障排查 | 一键获取相关日志、指标、链路信息 |
| 知识查询 | 快速检索历史故障处理方案 |
| 自动化 | 自动执行常见修复操作 |

### 典型语录

> "一个故障可能涉及十几个系统，排查起来像大海捞针。"
> 
> "如果能有一个工具帮我自动关联相关日志和指标，能节省一半时间。"
> 
> "很多问题以前处理过，但就是找不到当时的记录。"

## 画像三：运维工程师（日常用户）

### 基本信息

| 属性 | 描述 |
|------|------|
| **姓名** | 王芳（化名） |
| **职位** | 运维工程师 / NOC工程师 |
| **年龄** | 25-30岁 |
| **工作年限** | 2-5年 |
| **技术栈** | Linux、监控工具、基础脚本 |
| **企业规模** | 500-5000人企业 |
| **行业** | 金融、制造业、政务 |

### 个人背景

王芳是一位初级运维工程师，负责日常监控和告警处理。她有一定的技术基础，但对复杂系统的理解有限，遇到问题需要请教资深同事。她希望快速成长，但缺乏系统性的学习路径。

### 典型工作场景

- **监控值守**：查看监控大盘，处理告警
- **故障升级**：无法处理的问题升级给资深同事
- **日常巡检**：系统健康检查、日志审查
- **工单处理**：处理ITSM工单
- **文档维护**：更新运维文档

### 核心痛点

| 痛点 | 具体描述 | 痛苦程度 |
|------|---------|---------|
| 经验不足 | 遇到复杂问题不知道如何下手 | ⭐⭐⭐⭐⭐ |
| 依赖他人 | 频繁请教资深同事，感觉不好意思 | ⭐⭐⭐⭐ |
| 学习困难 | 缺乏系统性学习路径，成长缓慢 | ⭐⭐⭐⭐ |
| 告警疲劳 | 大量无效告警，难以区分优先级 | ⭐⭐⭐⭐ |
| 压力大 | 担心处理不当造成更大问题 | ⭐⭐⭐ |

### 目标与动机

**主要目标**：
- 独立处理更多问题，减少对资深同事的依赖
- 快速学习运维知识，提升技术能力
- 减少告警噪音，专注于有价值的工作
- 获得认可，实现职业成长

**使用动机**：
- AI辅助决策，增强处理问题的信心
- 学习资深工程师的经验
- 减少重复性工作，提升效率

### 使用场景

| 场景 | 具体需求 |
|------|---------|
| 告警处理 | 明确告警优先级和处理建议 |
| 故障排查 | 逐步引导的排查流程 |
| 知识学习 | 快速获取相关知识和案例 |
| 操作执行 | 安全的自动化操作 |

### 典型语录

> "遇到没见过的问题就慌，不知道从哪里开始排查。"
> 
> "如果能告诉我这个告警是什么意思、该怎么处理就好了。"
> 
> "不想每次都问前辈，但自己又处理不了。"

## 用户画像应用指南

### 产品设计指导
- **核心用户**：SRE工程师，优先满足其需求
- **决策用户**：运维经理，提供ROI证明材料
- **日常用户**：运维工程师，降低使用门槛

### 营销策略指导
- **决策链**：运维经理推荐 → CTO/CIO审批
- **价值主张**：效率提升、成本节约、风险降低
- **案例选择**：同行业、同规模企业案例

### 服务策略指导
- **培训重点**：运维工程师，提供详细使用指南
- **技术支持**：SRE工程师，提供深度技术交流
- **客户成功**：运维经理，定期汇报效果数据
