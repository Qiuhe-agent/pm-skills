# kc-builder (知识城邦建造师) PRD

## Problem Statement

个人知识管理目前面临以下痛点：
1. **知识枯燥无趣**：Markdown文件和文件夹结构缺乏视觉吸引力，难以激发持续整理和回顾的动力
2. **知识关联性难以感知**：纯文本结构无法直观展示知识之间的内在联系和主题聚类
3. **时间维度缺失**：知识积累的过程和演进难以可视化，缺少历史感和成长反馈
4. **现有工具过于复杂**：主流知识管理工具（Notion、Obsidian等）功能繁多，但缺少能让知识"活起来"的展示方式
5. **没有发挥LLM潜力**：现有工具只是把LLM当作文本生成器，没有让LLM参与到
知识的创意可视化中

## Solution

**kc-builder** 是一个基于Claude Code的agent skill，通过大模型的创意能力，将构建一个可视化的"知识城邦"，方便个人进行知识管理和打造个人的知识库。

核心思路：

1. **LLM作为创意核心**：所有建筑创意、城邦规划、页面生成都由大模型主导
2. **知识→建筑映射**：每个知识文件对应城邦中一座独特的建筑，建筑名称和风格天马行空但与知识内容相关
3. **主题→区域映射**：相关主题的知识聚集在同一个区域
4. **时间线可视化**：按知识创建/修改时间生成时间线，展示知识成长历程
5. **静态HTML输出**：生成美观、可交互的单页HTML，可用浏览器直接打开
6. 通过和大模型对话的方式构建知识。
9. 尽可能地使用大模型地能力，非必要不是代码脚本。

## User Stories

1. **As a** 个人知识管理者, **I want** 将我的Markdown知识目录转化为知识城邦, **so that** 我的知识变得生动有趣，更有动力去整理和回顾

2. **As a** Claude Code用户, **I want** 通过skill命令调用kc-builder, **so that** 我可以在Claude Code中无缝使用这个功能

3. **As a** 知识创作者, **I want** 每个知识文件对应一座独特的建筑, **so that** 我可以直观地识别和区分不同的知识

4. **As a** 视觉学习者, **I want** 知识城邦有美观的HTML可视化页面, **so that** 我可以通过浏览器直观地浏览我的知识

5. **As a** 知识回顾者, **I want** 时间线展示知识的创建和修改历史, **so that** 我可以看到自己知识积累的成长历程

6. **As a** 探索者, **I want** 点击建筑可以查看对应的知识内容, **so that** 我可以在城邦中探索和阅读我的知识

7. **As a** 主题学习者, **I want** 相关主题的知识聚集在同一个区域, **so that** 我可以直观地看到知识之间的关联性

8. **As a** 用户, **I want** 建筑的名称和风格天马行空但与知识内容相关, **so that** 知识城邦既有趣又有意义

9. **As a** Claude Code用户, **I want** 多种工作模式（手动构建、自动监听、对话创作）, **so that** 我可以根据场景选择最方便的使用方式

10. **As a** 频繁更新知识的用户, **I want** 生成结果有缓存, **so that** 避免重复调用LLM，节省成本和时间

11. **As a** 新用户, **I want** 简单的目录结构就能开始使用, **so that** 我不需要复杂的配置就能快速体验

12. **As a** 知识管理者, **I want** 全局索引记录所有知识和建筑的映射, **so that** 我可以快速查找和管理我的知识城邦

13. **As a** 创意爱好者, **I want** LLM能根据知识内容生成独特的建筑描述, **so that** 每个建筑都有自己的故事和个性

14. **As a** 用户, **I want** 生成的HTML是单文件、无外部依赖, **so that** 我可以方便地分享和存档

15. **As a** 回顾者, **I want** 城邦布局考虑知识间的关联性, **so that** 相关的知识在地理上也相近

16. **As a** 调试者, **I want** 能看到LLM的思考过程, **so that** 我可以理解建筑和布局是如何生成的

17. **As a** 用户, **I want** 可以重新生成某个建筑或整个城邦, **so that** 我可以尝试不同的创意方案

18. **As a** 知识整理者, **I want** 添加新知识文件后能快速更新城邦, **so that** 我的城邦始终与知识库同步

19. **As a** 用户, **I want** 不需要手动配置建筑映射规则, **so that** LLM能智能理解知识内容并创意建筑

20. **As a** 长期用户, **I want** 知识城邦能随着知识增长而扩张, **so that** 城邦的发展反映我的知识成长

## Implementation Decisions

### 模块架构（LLM优先）

系统采用纯LLM驱动的设计，最大程度发挥大模型的创意能力：

1. **SkillOrchestrator（指挥中心）**
   - Claude Code Skill入口点
   - 协调各LLM模块的工作流
   - 支持三种工作模式：手动构建、文件监听、对话创作
   - 管理用户输入和输出展示

2. **KnowledgeArchitect（知识理解与建筑创意师）**
   - 读取Markdown文件和元数据（创建/修改时间）
   - LLM核心：理解知识内容，创意生成建筑名称、类型、描述、风格
   - 输出：建筑创意数据结构
   - System Prompt设计：强调创意、关联性、独特性

3. **CityPlanner（城邦规划师）**
   - 接收所有建筑创意数据
   - LLM核心：设计城邦布局，包括区域划分、建筑位置、道路广场
   - 考虑因素：知识主题、创建时间、内容关联性
   - 输出：城邦布局数据结构

4. **TimelineCurator（时间线策展人）**
   - 按知识创建/修改时间排序
   - LLM核心：生成时间线叙事文案
   - 输出：时间线数据结构

5. **SiteConstructor（HTML/CSS建造师）**
   - 接收城邦布局和时间线数据
   - LLM核心：生成完整的HTML/CSS/JavaScript单页应用
   - 包含功能：城邦可视化、时间线展示、点击交互、知识内容查看
   - 输出：静态HTML文件

6. **IndexMemory（索引记忆库）**
   - 维护全局索引JSON文件
   - 缓存LLM生成结果，避免重复调用
   - 记录知识文件→建筑的映射关系
   - 支持增量更新

### LLM交互设计

每个模块都有专门的System Prompt，通过结构化提示词引导LLM输出JSON格式的结果。使用ReAct模式让LLM能够反思和优化其创意。

### 接口设计

- Skill入口：`/kc build` - 手动构建城邦
- Skill入口：`/kc watch` - 监听文件变化自动更新
- Skill入口：`/kc create` - 对话式创建知识和建筑
- Skill入口：`/kc refresh [building-id]` - 重新生成指定建筑

## Testing Decisions

### 测试原则
- 测试外部行为，不测试LLM内部推理
- 测试数据结构的正确性，不测试创意内容的好坏
- 使用mock LLM响应进行确定性测试
- 集成测试使用真实LLM但限制调用次数

### 需要测试的模块

1. **SkillOrchestrator测试**
   - 测试命令解析和路由
   - 测试工作流编排
   - 测试错误处理

2. **KnowledgeArchitect测试**
   - 测试Markdown文件读取
   - 测试LLM响应解析
   - 测试建筑数据结构验证

3. **CityPlanner测试**
   - 测试建筑数据聚合
   - 测试布局数据结构验证
   - 测试区域分组逻辑

4. **TimelineCurator测试**
   - 测试时间排序
   - 测试时间线数据结构

5. **SiteConstructor测试**
   - 测试HTML文件生成
   - 测试HTML有效性（基本语法检查）

6. **IndexMemory测试**
   - 测试JSON读写
   - 测试缓存命中/未命中逻辑
   - 测试增量更新

### 测试数据策略
- 提供示例Markdown文件集合作为测试用例
- 预录制mock LLM响应用于快速测试
- 集成测试使用小型测试集合

## Out of Scope

以下功能不在本PRD范围内：

1. **多用户协作** - 当前聚焦个人用户，不考虑多人同时编辑
2. **云端同步** - 纯本地方案，不涉及云端存储和同步
3. **移动端优化** - 桌面优先，暂不考虑移动端适配
4. **复杂的权限系统** - 不需要用户认证和权限管理
5. **插件系统** - 不支持第三方插件扩展
6. **实时多人在线** - 不支持WebSocket或实时协作
7. **数据库集成** - 使用本地文件系统，不引入数据库
8. **高级搜索** - 基础索引即可，暂不考虑全文检索
9. **主题/皮肤切换** - 单个默认样式即可
10. **国际化多语言** - 中文优先，暂不考虑多语言支持

## Further Notes

### 设计理念
- **LLM优先**：能用LLM解决的就不用算法，最大限度发挥大模型的创意能力
- **极简主义**：核心功能要简单、直观，避免过度设计
- **可扩展性**：预留扩展点，但不提前实现不需要的功能
- **缓存友好**：合理设计缓存策略，平衡创意多样性和成本效率

### 成功指标
- 用户能够在5分钟内从零开始生成第一个知识城邦
- LLM调用次数优化到合理范围（单次生成控制在10次以内）
- 生成的HTML页面在现代浏览器中流畅运行
- 用户反馈"有趣"、"有创意"、"想继续使用"

### 风险与缓解
- **LLM输出不稳定**：使用结构化提示词+输出验证+重试机制
- **生成速度慢**：实现缓存机制，增量更新
- **成本过高**：优化提示词，减少token使用，提供成本控制选项
