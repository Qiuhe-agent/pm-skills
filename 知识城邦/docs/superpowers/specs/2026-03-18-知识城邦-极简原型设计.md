# 知识城邦 - 极简原型设计文档

**日期**: 2026-03-18
**状态**: 已确认
**版本**: MVP

## 概述

管理个人碎片化信息，打造个人知识库，最终创建个人数字孪生。

## 核心概念

- **空间维度**：用建筑展示领域知识
- **时间维度**：建筑随时间"生长"，不同日期展示不同形态
- **风格参考**：Minecraft
- **交互方式**：对话式存储，HTML 可视化展示

## 目标

先做能跑的最小版本，快速验证概念。

## 2. 数据层设计

### 2.1 文件结构

```
知识城邦/
├── data/
│   ├── knowledge.json          # 知识条目
│   ├── tags.json               # 标签体系
│   ├── state.json              # 对话状态
│   └── relations.json         # 关联关系（后期加）
```

### 2.2 knowledge.json

知识条目存储格式：

```json
[
  {
    "id": "uuid-1",
    "content": "今天思考了知识城邦的架构，觉得应该先从极简原型开始...",
    "title": "知识城邦架构思考",
    "tags": ["知识管理", "项目规划"],
    "type": "note",
    "created_at": "2024-06-15T10:30:00",
    "updated_at": "2024-06-15T10:30:00"
  }
]
```

字段说明：
- `id`: 唯一标识符（UUID）
- `content`: 知识内容
- `title`: 标题（可选）
- `tags`: 标签数组
- `type`: 类型（note/article/link/...）
- `created_at`: 创建时间
- `updated_at`: 更新时间

### 2.3 tags.json

标签体系格式：

```json
{
  "知识管理": {
    "count": 5,
    "color": "#4CAF50",
    "building_type": "library"
  },
  "项目规划": {
    "count": 3,
    "color": "#2196F3",
    "building_type": "office"
  }
}
```

字段说明：
- `count`: 该标签下的知识数量
- `color`: 可视化颜色
- `building_type`: 建筑类型（library/office/garden/warehouse）

**building_type 可选值**：
- `library`: 图书馆（知识管理、学习、阅读类标签）
- `office`: 办公楼（项目、工作、规划类标签）
- `garden`: 花园（想法、灵感、创意类标签）
- `warehouse`: 仓库（资料、参考、收集类标签）

#### tags.json 初始化与更新规则

**初始化**：
- 初始时 tags.json 为空对象 `{}`
- 当新笔记添加标签时，自动在 tags.json 中创建该标签条目

**自动生成规则**：
- `count`: 自动统计该标签下的知识数量
- `color`: 从预设颜色列表中按顺序分配（#4CAF50, #2196F3, #FF9800, #E91E63, #9C27B0...）
- `building_type`: 根据 4.3 节的标签→建筑类型映射规则自动确定

**更新时机**：
- 添加/删除标签时更新 count
- 新增标签时分配 color 和 building_type

### 2.4 state.json

对话状态存储格式：

```json
{
  "last_knowledge_id": "uuid-1"
}
```

字段说明：
- `last_knowledge_id`: 最近一条操作的笔记 ID

### 2.5 relations.json（第二阶段，MVP 不包含）

关联关系格式：

```json
[
  {
    "from": "uuid-1",
    "to": "uuid-2",
    "strength": 0.8,
    "reason": "都提到了知识管理"
  }
]
```

## 3. 对话交互层设计

### 3.1 对话示例

| 用户说 | 系统理解 | 回应 |
|-------|---------|------|
| "存个笔记：今天的想法是..." | 存储笔记 | "已保存！添加了 1 条笔记。" |
| "标题叫'知识城邦想法'，加标签：项目" | 修改上一条 | "已更新！标题和标签已设置。" |
| "给这篇加个标签：想法" | 给最近一条加标签 | "标签'想法'已添加。" |
| "搜索关于知识管理的" | 搜索 | "找到了 3 条相关笔记：..." |
| "打开知识城邦" | 打开 HTML | "正在打开知识城邦可视化..." |

### 3.2 对话流程设计

- 系统记住"最近一条"笔记（即最后创建或修改的那条），方便连续操作
- 用简单关键词识别意图，按优先级匹配
- 先不做复杂 NLP

#### "最近一条"笔记状态管理

**存储方案**：
- 在 `data/` 目录下创建 `state.json` 文件
- 格式：`{ "last_knowledge_id": "uuid-xxx" }`

**更新时机**：
- 创建新笔记后，更新为新笔记的 id
- 修改笔记（标题、标签）后，更新为该笔记的 id
- 读取操作不更新

**生命周期**：
- "最近一条"永久保存，直到被新操作覆盖

### 3.3 关键词识别（优先级从高到低）

意图识别规则：

1. **打开可视化**：如果消息包含"打开"且包含"知识城邦"或"可视化"或"展示"
2. **搜索**：如果消息以"搜索"或"找"或"查找"开头
3. **加标签**：如果消息包含"标签"或"加标签"
4. **修改/标题**：如果消息以"标题"开头
5. **存储笔记**：如果消息包含"存"或"保存"或"笔记"或"记录"，或者上述都不匹配但有实质内容

#### 标题提取规则（用于规则 4）

- 从消息中提取引号（"" 或 ''）中的内容作为标题
- 如果没有引号，取"标题"后面的所有内容作为标题
- 示例：
  - "标题叫'知识城邦想法'" → 标题 = "知识城邦想法"
  - "标题：关于AI的思考" → 标题 = "关于AI的思考"

#### 加标签提取规则（用于规则 3）

- 从消息中提取"标签："或"标签"后面的内容
- 多个标签用逗号、顿号或空格分隔
- 示例：
  - "加标签：项目，想法" → 标签 = ["项目", "想法"]
  - "标签：知识管理 学习" → 标签 = ["知识管理", "学习"]

**"实质内容"定义**：
- 消息长度 > 10 个字符，且
- 不是纯疑问句（不以"？"或"?"结尾），且
- 不是简单的确认/否定（如"好的"、"对"、"不"、"是的"）

**示例**：
- "今天思考了一下这个问题" → 有实质内容，存为笔记
- "存一下" → 匹配关键词，提示用户："你想存什么内容？"（不创建空笔记）
- "好的" → 无实质内容，不存储
- "对吗？" → 疑问句，不存储

**歧义处理**：
- 只处理单一意图，一条消息只做一件事
- 如果有多个意图，按优先级匹配第一个
- 如果无法识别，询问用户："你想做什么？存笔记、搜索、还是打开可视化？"

## 4. 可视化层设计

### 4.1 布局设计

```
┌─────────────────────────────────────────────────────┐
│  🏛️ 知识城邦                    [2D] [📅 2024-06-15 ▼] │
├─────────────────────────────────────────────────────┤
│                                                     │
│    ┌─────────────┐         ┌─────────────┐        │
│    │   图书馆    │         │   办公楼    │        │
│    │  知识管理   │         │  项目规划   │        │
│    │   (5条)     │         │   (3条)     │        │
│    └─────────────┘         └─────────────┘        │
│                                                     │
│              ┌─────────────┐                       │
│              │   花园      │                       │
│              │    想法      │                       │
│              │   (10条)     │                       │
│              └─────────────┘                       │
│                                                     │
├─────────────────────────────────────────────────────┤
│  时间轴：[2024-01] [2024-02] [2024-03] [2024-04] │
└─────────────────────────────────────────────────────┘
```

### 4.2 交互设计

- **点击建筑** → 弹窗显示该标签下的笔记列表
- **日期选择（下拉框 + 时间轴按钮）** → 重新计算该时间点前的建筑大小
  - 顶部下拉框：选择具体月份（如"2024-06"）
  - 底部时间轴：最近 12 个月的快捷按钮，点击快速切换
- **鼠标悬停** → 显示标签详情

### 4.3 建筑映射

#### 标签 → 建筑类型映射规则

**自动映射逻辑**：

| 建筑类型 | 关键词匹配 |
|---------|---------|
| 图书馆 | 标签包含"知识"、"学习"、"阅读"、"笔记"、"读书" |
| 办公楼 | 标签包含"项目"、"工作"、"规划"、"计划"、"任务" |
| 花园 | 标签包含"想法"、"灵感"、"创意"、"思考" |
| 仓库 | 标签包含"资料"、"参考"、"收集"、"链接" |

**关键词匹配优先级**：
- 按表格顺序从上到下匹配（图书馆 → 办公楼 → 花园 → 仓库）
- 第一个匹配成功的建筑类型即为结果
- 如果标签同时匹配多个关键词，以最先匹配的为准

**示例**：
- 标签"知识项目" → 匹配"知识" → 图书馆
- 标签"工作灵感" → 匹配"工作" → 办公楼
- 标签"思考资料" → 匹配"思考" → 花园

**默认规则**：
- 如果不匹配上述关键词，默认为"图书馆"
- 用户可后续手动修改（MVP 暂不支持手动修改）

#### 多标签建筑映射规则

当一个知识条目有多个标签时：
- **每个标签一个建筑**：每个标签独立显示一个建筑
- 知识计数：如果一个知识有多个标签，它会被计入每个相关标签的 count
- 示例：
  - 笔记 A 有标签 ["知识管理", "项目"]
  - 笔记 A 会同时出现在"图书馆"和"办公楼"中
  - 两个建筑的 count 都会 +1

#### 建筑大小计算逻辑

**建筑宽度**：
```
基础宽度 = 80px
每增加 1 条知识，宽度增加 10px
最大宽度 = 200px
```

**建筑高度**：
```
基础高度 = 60px
每增加 1 条知识，高度增加 8px
最大高度 = 150px
```

**时间筛选逻辑**：
- 选择日期时，只统计 `created_at` <= 所选日期的知识
- 重新计算每个标签的 `count` 和建筑大小

### 4.4 建筑布局算法

建筑按简单网格排列：
- 从左到右，从上到下
- 每行最多 3 个建筑
- 建筑间距：30px
- 按标签知识数量从多到少排序

### 4.5 技术实现

- 纯 HTML + CSS + JavaScript（无框架）
- **使用 DOM + CSS 绘制 2D 视图**（不使用 Canvas）
  - 建筑用 `<div>` 元素表示
  - 布局用 CSS Flexbox/Grid
  - 这样更容易实现点击、悬停等交互
- 直接 fetch 本地 JSON 文件

#### 搜索功能实现

- 搜索范围：标题、内容、标签
- 匹配方式：子字符串匹配（不区分大小写）
- 结果排序：按创建时间倒序

#### 日期选择器精度

- 按月选择（如"2024-06"）
- 显示该月及之前的所有知识
- 时间轴显示最近 12 个月

#### HTML 可视化打开方式

- Skill 收到"打开知识城邦"等指令后：
  1. 确保 `public/index.html` 文件存在
  2. 使用系统默认浏览器打开该 HTML 文件
  3. HTML 文件通过 `fetch()` 读取同目录下的 `../data/knowledge.json` 和 `../data/tags.json`
  4. 数据文件路径使用相对路径，确保在本地文件系统中也能正常工作

## 附录

### A. 风险与应对

| 风险 | 应对 |
|-----|------|
| 原型太简陋，体验不好 | 先做能用，再优化体验 |
| 后期需要重写代码 | 保持代码简单，重写成本低 |
| 概念验证失败 | 快速失败，损失小 |

### B. 成功标准

- 可以正常存储和检索笔记
- 可以通过对话交互
- 可视化能正常展示
- 自己愿意每天使用

### C. MVP 验收标准

| 检查项 | 验收标准 |
|-------|---------|
| 文件夹结构 | 已创建完整的文件夹结构（skill/, data/, public/, docs/） |
| Skill 对话入口 | 可以通过对话与系统交互 |
| 笔记存储 | 可以通过对话创建新笔记，笔记正确保存到 knowledge.json |
| 标签系统 | 可以给笔记添加标签，tags.json 自动更新 |
| 搜索功能 | 可以通过关键词搜索笔记，匹配标题、内容、标签 |
| 2D 可视化 | HTML 页面可以正常显示建筑（用 div 绘制） |
| 时间切换 | 可以通过月份下拉框和时间轴按钮切换时间，建筑大小相应变化 |
| 标签初始化 | 新标签自动分配颜色和建筑类型 |
| 多标签展示 | 一个笔记有多个标签时，会在多个建筑中显示 |
| 打开可视化 | 可以通过对话打开 HTML 可视化页面 |
