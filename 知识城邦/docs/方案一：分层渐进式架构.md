# 方案一：分层渐进式架构

## 核心理念

清晰分层，每一层可以独立演进。以数据为基础，通过知识处理层构建关联，再通过城邦生成层转化为可视化概念，最终在 HTML 中展示。

## 架构图

```
┌─────────────────────────────────────────┐
│   对话交互层 (Skill)                    │
│   - 自然语言输入信息                  │
│   - 命令控制                           │
└──────────────┬──────────────────────────┘
┌──────────────▼──────────────────────────┐
│   知识处理层 (Subagents)                 │
│   - 分类打标签                          │
│   - 构建知识关联                        │
│   - 时间维度管理                        │
└──────────────┬──────────────────────────┘
┌──────────────▼──────────────────────────┐
│   城邦生成层                             │
│   - 知识 → 建筑映射规则                 │
│   - 时间轴建筑生长计算                  │
└──────────────┬──────────────────────────┘
┌──────────────▼──────────────────────────┐
│   可视化层 (HTML)                        │
│   - 2D 俯视图                           │
│   - 3D 视图 (Three.js)                  │
│   - 时间切换器                          │
└─────────────────────────────────────────┘
┌─────────────────────────────────────────┐
│   数据层 (本地文件)                      │
│   - 知识条目存储                        │
│   - 关联关系存储                        │
│   - 城邦状态快照                        │
└─────────────────────────────────────────┘
```

## 详细设计

### 1. 对话交互层 (Skill)

**职责**：
- 接收用户的自然语言输入
- 解析用户意图（存储、搜索、查询、修改等）
- 调用下层子代理完成任务
- 以自然语言返回结果

**主要功能**：
- "帮我存个笔记：关于AI的想法..."
- "给这篇文章打个标签：产品经理"
- "搜索一下关于'知识管理'的内容"
- "切换到 2024-06-01 的城邦视图"

---

### 2. 知识处理层 (Subagents)

**职责**：
- 处理知识的结构化
- 建立知识之间的关联
- 管理时间维度信息

**子代理划分**：

#### 分类打标签代理
- 自动识别内容类型
- 建议标签
- 维护标签体系

#### 关联构建代理
- 识别内容之间的语义关联
- 构建知识图谱
- 推荐相关内容

#### 时间管理代理
- 记录知识创建/修改时间
- 管理时间轴里程碑
- 计算不同时间点的城邦状态

---

### 3. 城邦生成层

**职责**：
- 将抽象的知识映射为具象的城邦概念
- 计算建筑在不同时间点的生长状态

**知识 → 建筑映射规则（可配置）**：

| 知识属性 | 城邦元素 | 说明 |
|---------|---------|------|
| 知识领域 | 区域/区块 | 不同领域在不同位置 |
| 知识数量 | 建筑大小 | 知识越多，建筑越大 |
| 知识重要性 | 建筑高度 | 越重要，建筑越高 |
| 知识类型 | 建筑类型 | 笔记→图书馆，项目→工厂，灵感→花园 |
| 知识关联 | 道路/桥梁 | 关联越强，道路越宽 |
| 创建时间 | 建造阶段 | 地基→框架→内部→完工 |

**时间生长算法**：
- 每个知识条目贡献建筑的一部分
- 按时间顺序累加，形成生长过程
- 关键节点（如里程碑知识）触发建筑形态变化

---

### 4. 可视化层 (HTML)

**技术栈**：
- 2D：SVG 或 Canvas
- 3D：Three.js
- 交互：原生 JavaScript

**视图模式**：

#### 2D 俯视图
- 类似游戏地图
- 不同颜色代表不同领域
- 鼠标悬停显示信息
- 可缩放、平移

#### 3D 视图
- 可旋转、可进入建筑
- Minecraft 风格方块美学
- 时间动画展示生长过程

#### 时间控制器
- 时间轴滑块
- 关键日期书签
- 播放/暂停动画

---

### 5. 数据层

**文件结构**：
```
知识城邦/
├── data/
│   ├── knowledge/          # 知识条目
│   │   ├── 2024-01-01/
│   │   ├── 2024-01-02/
│   │   └── ...
│   ├── tags/               # 标签体系
│   ├── relations/          # 关联关系
│   └── snapshots/          # 城邦状态快照
└── exports/                # 导出版本
    └── minecraft/          # Minecraft 存档
```

**数据格式**：JSON（易读、易版本控制）

---

## 优点

- 每层职责清晰，易于理解和维护
- 可以先做简单的映射规则，后续再优化
- 可视化层可以独立演进（先 2D，再加 3D）
- 适合用 Claude Code 的 Skill + Subagent 模式
- 可以借鉴 Minecraft 但不被其限制

## 缺点

- 初期需要设计好各层接口
- 需要更多的前期设计工作
